# Civic Complaint Copilot

A comprehensive guide for building a production-ready civic issue reporting web application. This skill provides the required architecture, AI prompt chains, and user flow for a system where citizens can report issues (potholes, garbage, etc.) and generate formal, actionable complaints.

## 1. System Architecture & Component Structure

When building the application, adhere to the following modular React structure rather than a single-file monolith:

```
src/
├── App.jsx                    # Root component, routing setup
├── api/
│   └── claude.js              # Proxy wrapper for secure AI calls
├── components/
│   ├── Nav.jsx                # Global navigation
│   ├── Tag.jsx                # UI for displaying severity, status, and issue type
│   ├── ErrBox.jsx             # Reusable error boundaries/messages
│   └── Processing.jsx         # Loading state with possible abort/cancel
├── screens/
│   ├── Home.jsx               # Landing page & calls to action
│   ├── Report.jsx             # Form for GPS, image upload, and text description
│   ├── Result.jsx             # Display generated complaint, PDF export, and submit buttons
│   └── Dashboard.jsx          # List of user's past complaints and statuses
├── config/
│   ├── authorities.js         # AUTHORITY_MAP (e.g., 'pothole' -> 'Municipal Dept of Roads')
│   ├── theme.js               # Global UI constants, colors
│   └── constants.js           # Valid issue types and severities
├── hooks/
│   ├── useComplaints.js       # CRUD operations for complaints (localStorage sync)
│   └── useGeolocation.js      # GPS detection hook with reverse geocoding
├── utils/
│   ├── id.js                  # ID generation (e.g. `genId()`)
│   ├── date.js                # Date formatting utilities
│   ├── pdf.js                 # PDF generation logic (via jsPDF)
│   └── parseJSON.js           # Robust JSON extraction from AI responses
└── styles/
    └── index.css              # Global styles and utility classes
```

## 2. 5-Screen Navigation Flow

Implement a clear, linear navigation flow to minimize user friction:

1. **Home**: Landing screen with "Report an Issue" button and quick stats.
2. **Report**: The intake form. Needs fields for issue description, photo upload (base64 compressed limit), and auto-detected GPS location.
3. **Processing**: An interstitial loading screen shown while the AI prompt chain is executing. Must include a timeout and cancel option.
4. **Result**: Displays the AI-classified output (issue type, severity) and the generated formal complaint letter. Offers actions: "Edit", "Submit via Email/WhatsApp", "Download PDF".
5. **Dashboard**: A persistent list of all complaints generated by the user, showing their ID, date, status, and severity.

## 3. Data Flow & Features

### The 2-Step AI Prompt Chain

To ensure high-quality, structured data, split the AI task into two separate operations:

**Step 1: Classification & Fact Extraction (JSON)**
Prompt the AI to act as a triage officer. It must return pure JSON (no markdown wrapping) containing:
- `issue_type` (e.g., "pothole", "garbage", "streetlight")
- `severity` ("low", "medium", "high") based on public risk
- `issue_summary` (1-2 sentence brief)

*Crucial Implementation Detail:* Use a robust regex to parse the JSON and strip markdown backticks, because the LLM might wrap the output with prose.

**Step 2: Formal Complaint Generation (Text)**
Prompt the AI to act as a legal/administrative writer. Pass in the classification from Step 1, the user's description, and GPS location to generate a formal, polite, but firm grievance letter addressed to the relevant authority.

### Authority Mapping & Config
Map `issue_type` directly to the correct department so the user doesn't have to guess. Example:
```javascript
export const AUTHORITY_MAP = {
  pothole: "Municipal Dept of Roads",
  garbage: "Sanitation & Solid Waste Dept",
  streetlight: "Electricity Board",
  noise: "Local Police Station",
  other: "Municipal Corporation Office"
};
```

### GPS Detection & Reverse Geocoding
Don't just use raw coordinates for the complaint (e.g., `28.61390, 77.20900`), as they are useless for formal letters.
- Use `navigator.geolocation.getCurrentPosition`.
- Fall back to reverse geocoding via the free Nominatim OpenStreetMap API to get a human-readable street address.

### Complaint ID Generation
Every complaint needs a unique tracking ID. Use a simple generator like `CCC-` followed by an alphanumeric string or timestamp hex (e.g., `CCC-A72B-91X`).

### PDF Export
Integrate `jsPDF` to allow users to download a physical copy of their formal complaint. Make sure to bundle `jspdf` as an npm dependency rather than loading it dynamically via a CDN to ensure offline capability and speed.

### Persistent Storage
Save all generated complaints bridging sessions.
- Use `window.storage` with a `localStorage` fallback as the persistence layer.
- Ensure the CRUD hook (`useComplaints`) loads data into React state on mount and syncs it back on every write/delete.

## 4. Security & Production Checklist

Before finalizing the app, ALWAYS ensure:
1. **API Keys are Server-Side**: The AI API key must never be exposed in client code. Route requests through a minimal serverless proxy (e.g., Serverless Function or Node proxy).
2. **Prevent memory leaks**: If generating Blob URLs for images, make sure to revoke the previous object URL on a new upload (`URL.revokeObjectURL()`).
3. **Form Validation**: Ensure Location and Issue Description have standard inline validation before allowing form submission to save API usage tokens and prevent malformed data.
